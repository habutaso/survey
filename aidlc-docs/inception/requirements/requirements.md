# 要件定義書: 住家被害認定調査 効率化アプリ

## 1. インテント分析サマリー

- **ユーザー要求**: タブレットで画像を撮影しながら被害度を入力し、最後に被害度を自動計算して、その家屋が何段階の被害度かを表示する「住家被害認定調査の調査効率化アプリ」。
- **要求タイプ**: 新機能（New Feature）。既存 CATAPULT テンプレート上に新ドメインを構築。
- **スコープ見積り**: システム全体（client/server 双方に新ドメインを追加、既存デモ Task/User ドメインを置換）。
- **複雑度見積り**: Complex（被害度の自動計算ロジックが内閣府運用指針に準拠する必要があり、第1次・第2次調査の双方とその関連管理を含む専門的ドメイン）。
- **対象プロジェクト種別**: Brownfield（CATAPULT テンプレート）。
- **準拠基準**: 内閣府「災害に係る住家の被害認定基準運用指針」（`docs/references/` の公式資料、bousai.go.jp 取得、令和版）。

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| 調査 (Survey) | 1家屋に対する1回の被害認定調査。住所・家屋情報で識別する集約。調査区分（第1次／第2次）を持つ。 |
| 調査区分 (Survey Type) | 第1次調査 または 第2次調査 の別。 |
| 損害割合 | 住家全体の経済的被害の割合（％）。被害度区分の判定基準。 |
| 被害度区分 | 損害割合に基づく6区分（全壊／大規模半壊／中規模半壊／半壊／準半壊／準半壊に至らない（一部損壊））。 |
| 第1次調査 | 外観目視による調査（外力・流失判定／傾斜判定／浸水深判定）。 |
| 第2次調査 | 内部に立ち入り、外部部位（屋根・基礎・外壁）＋内部部位（天井・床・内壁・建具・設備 等）の損傷率を判定し、部位別損傷率×構成比で住家全体の損害割合を算出する詳細調査。第1次調査の確定後に、その再調査として実施する。 |
| 再調査 | 第1次調査の確定結果に対し、より詳細に判定し直す目的で実施する第2次調査。元の第1次調査に関連付けて履歴を残す。 |
| 部位 | 判定対象となる住家の構成要素（屋根・柱(傾斜)・外壁・基礎・天井・床・内壁・建具・設備 等）。 |
| 構成比 | 各部位が住家全体に占める標準的な経済価値の割合。運用指針の標準値（固定）を使用。 |
| 浸水深 | 床上／床下からの浸水の深さ（cm）。水害の被害度判定に用いる実測値。 |
| 傾斜 | 外壁・柱の傾斜（実測）。損害割合の加算要素。 |
| ロール | 調査員 / 管理者 / 閲覧者。 |

## 3. 機能要件 (Functional Requirements)

### 3.1 調査管理

- **FR-01**: システムは1家屋を単位とする調査を新規作成できること。調査には家屋識別情報（住所、家屋番号等の建物属性）を登録する。（出典: A7=A）
- **FR-02**: システムは被災者の個人情報（氏名・住所等）を調査に関連付けて保持できること。罹災証明書発行との連携を想定する。（出典: B5=A）
- **FR-03**: システムは調査位置（GPS による緯度経度）を記録できること。（出典: B4=A）
- **FR-04**: 調査は以下の状態を持ち、状態遷移は権限に従うこと: 下書き → 提出 → 承認 → 確定。（出典: A8=B）
- **FR-05**: 確定済みの調査は内容を変更できないこと（不変。再調査は新規/差戻しで対応）。
- **FR-06**: 各調査は調査区分（第1次／第2次）を持つこと。（出典: CQ-S2, 2次調査追加）
- **FR-07**: 第2次調査は、確定済みの第1次調査に対する**再調査**として作成し、元の第1次調査に関連付けて保持すること（双方の記録・経緯を履歴として残す）。第1次・第2次は別々の調査記録として管理する。（出典: CQ-S1=A, CQ-S3=A）
- **FR-08**: 第2次調査の開始（再調査）は、対象の第1次調査が確定済みであることを前提とし、管理者または調査員が開始できること。（出典: CQ-S3=A）
- **FR-09**: 同一家屋に第1次・第2次の双方の結果が存在する場合、システムは両方の被害度を併記して表示・記録し、いずれを正式とするかを**管理者が選択・確定**できること。（出典: CQ-S4=B）

### 3.2 被害度の入力と画像撮影

- **FR-10**: 調査員はタブレットのブラウザ／PWA から、部位ごとに被害状況を入力できること。（出典: B3=A）
- **FR-11**: 調査員は実測値（浸水深 cm、傾斜、各部位の損傷率等）を数値入力でき、アプリが損害割合・被害度区分を自動計算すること。（出典: CQ5=A）
- **FR-12**: 調査員は部位ごとに画像を撮影・添付できること（1部位に複数枚可）。（出典: A6=C）
- **FR-13**: 調査員は調査全体に対する全体写真も撮影・添付できること。（出典: A6=C）
- **FR-14**: 画像は撮影と入力を並行して行えるよう、入力フローの各部位ステップに紐付けて管理すること。
- **FR-15**: システムは調査入力を**ガイド付きステップ進行（ウィザード形式）**で案内すること。調査員は定義された手順に沿って順に入力でき、各ステップで保存・中断・再開できる。（出典: 追加要望 2026-06-12）
- **FR-16**: システムは**進捗パンくず（ステッパー）**を常時表示し、現在のステップ・完了済みステップ・残ステップを可視化すること。可能な範囲で既存ステップへ移動できる。（出典: 追加要望）
- **FR-17**: 標準のステップ順序は次を基本とすること: **住民情報入力 → 第1次/第2次調査の選択 → 調査入力 → （該当時）2階部分の割合調整 → 計算結果確認**。第2次調査では内部部位入力を含む。条件（単層住家等）により該当しないステップはスキップ表示する。（出典: 追加要望, FR-06〜09 と整合）
- **FR-18**: システムは**ローカルファースト保存**を行うこと。調査の入力内容（住民情報・実測値・損傷率・進捗状態等）および撮影画像を、オフラインでもブラウザのローカルストレージ（IndexedDB）に保持し、各ステップで保存・中断・再開できる。アプリ再読込・端末スリープを跨いでもローカルデータが維持されること。（出典: AD2-FU=A）
- **FR-19**: システムは**提出時の一括同期**を行うこと。調査員が「提出（下書き→提出）」を実行した時点で、IndexedDB に保持した入力内容・画像をサーバへ一括送信する。送信は再試行可能とし、ネットワーク不通時は提出をキューイングしてオンライン復帰後に同期、失敗時はローカルデータを保持してエラーを通知すること。同期完了の確認後にローカルの PII・画像を消去する方針は §6（SECURITY-01）に従う。（出典: AD2-FU=A, AD2-FU2=A）

### 3.3 被害度自動計算（ドメインの中核）

対象は**水害のみ**。第1次調査の全判定経路（外力・流失／傾斜／浸水深）と、第2次調査（外部＋内部の全部位による部位別×構成比）の双方を実装する。（出典: A1=B, CQ1=A, CQ3=B, 2次調査追加 CQ-S1〜S4）

**第1次調査の判定（外観目視）:**

- **FR-20**: システムは**外力・流失等による判定**を行うこと（住家流失・地盤洗掘・基礎流失・全部の階の天井までの浸水等に該当する場合は全壊と判定）。
- **FR-21**: システムは**傾斜による判定**を行うこと（外壁・柱の傾斜の実測値から損害割合への寄与を算出）。
- **FR-22**: システムは**浸水深による判定**を行うこと（床上浸水深から損害割合を算出）。

**第2次調査の判定（内部立入・部位別）:**

- **FR-23**: システムは**部位別損傷率 × 構成比**による判定（第2次調査）を行うこと。判定対象は外部部位（屋根・基礎・外壁）＋内部部位（天井・床・内壁・建具・設備 等）の**全部位**とし、各部位の損傷率に運用指針の標準構成比（固定値）を乗じて合計し、住家全体の損害割合を算出する。（出典: A5=A, CQ3=B, CQ-S2=A）

**共通（区分判定・出力）:**

- **FR-24**: システムは算出した損害割合（％）を、公式6区分（全壊≥50%／大規模半壊40–50%未満／中規模半壊30–40%未満／半壊20–30%未満／準半壊10–20%未満／準半壊に至らない<10%）へ正しくマッピングし、被害度区分を判定すること。（出典: CQ2=A）
- **FR-25**: 計算ロジックは内閣府運用指針（`docs/references/` の水害編・調査票・記入の手引き）に準拠すること。構成比・浸水深-損害割合の換算表は運用指針の標準値を用いる。
- **FR-26**: 計算結果として、損害割合（％）と被害度区分の双方を表示・記録すること。
- **FR-27**: 計算は入力値から決定論的に再現可能であること（同一入力 → 同一結果）。
- **FR-28**: 複数階建ての住家について、**階ごとの床面積比率（例: 1階・2階の按分）を調整・反映**して部位別損害割合を算定できること。階の割合調整は調査入力フローの一ステップとして提供し、運用指針に準拠する。（出典: 追加要望「2階部分の割合調整」, 詳細は Functional Design で確定）

### 3.4 結果表示・出力

- **FR-30**: システムは調査終了時に、その家屋の被害度区分（6区分）と損害割合を画面表示すること。（出典: 元要求, CQ2=A）
- **FR-31**: システムは調査結果を PDF として出力できること（調査票・判定結果を含む）。（出典: A9=D）
- **FR-32**: システムは調査結果（複数件）を CSV／Excel としてエクスポートできること。（出典: A9=D）
- **FR-33**: PDF 出力は内閣府の住家被害認定調査票の項目構成を参考にすること（`docs/references/` の調査票様式）。

### 3.5 認証・ロール・権限

- **FR-40**: システムは利用者を認証すること（既存 Cognito 認証を踏襲）。（出典: C1=A）
- **FR-41**: システムは3つのロールを持つこと: 調査員 / 管理者 / 閲覧者。（出典: A10=C）
- **FR-42**: 権限割当は以下とすること（出典: CQ4=A）:
  - 調査員: 調査の作成・入力・提出
  - 管理者: 提出された調査の承認・確定、全調査の管理
  - 閲覧者: 調査の参照のみ（自治体担当者等）
- **FR-43**: すべての調査リソースアクセスは、所有権・ロールに基づくサーバ側認可チェックを行うこと（オブジェクトレベル認可）。

## 4. 非機能要件 (Non-Functional Requirements)

- **NFR-01 (端末/UX)**: タブレットのブラウザ／PWA で快適に動作すること。現場での片手操作・屋外視認性を考慮する。（出典: B3=A）
- **NFR-02 (オフライン/ローカルファースト)**: 本リリースで**オフライン入力・ローカルファーストを正式採用**する。通信が不安定／不通の現場でも、調査員は入力内容・撮影画像をブラウザのローカルストレージ（IndexedDB）に保持して継続入力でき、**「提出（下書き→提出）」操作時にサーバへ一括同期**する。同期は再試行可能とし、同期失敗時はローカルデータを保持して喪失を防ぐこと。ローカルに保持する PII・画像の保護は §6（SECURITY-01）に従う。（出典: B1 → AD2-FU=A で改定, 同期トリガ AD2-FU2=A）
- **NFR-03 (規模/性能)**: 中規模（数百人規模の同時利用）に耐えること。一覧・検索は応答性を確保する。（出典: B2=B）
- **NFR-04 (正確性)**: 被害度計算は運用指針に準拠し、決定論的かつ検証可能であること。計算ロジックはプロパティベーステストで不変条件を検証する（D3=A, 後述 §7）。
- **NFR-05 (セキュリティ)**: セキュリティ・ベースライン（SECURITY-01〜15）を満たすこと。個人情報を扱うため特に重要（§6 参照）。（出典: D1=A, B5=A）
- **NFR-06 (技術構成)**: 既存 CATAPULT 構成（Next.js / Fastify / frourio + aspida / Prisma + PostgreSQL / Cognito / S3 / DDD・関数型・DI）をそのまま踏襲する。（出典: C1=A）
- **NFR-07 (アクセシビリティ)**: 画面はアクセシビリティに配慮すること（入力フォーム・コントラスト等）。
- **NFR-08 (監査性)**: 状態遷移（提出・承認・確定）および個人情報・確定データの変更は、実施者・日時を含めて監査可能に記録すること（SECURITY-13/14）。
- **NFR-09 (保守性/テスト容易性)**: 全関数を DI 可能とし（velona）、ドメインロジックはフレームワーク非依存の純粋関数として実装すること（DDD ガイドライン準拠）。

## 5. データ要件

保持する主なデータ（詳細スキーマは設計段階で確定）:

- **調査 (Survey)**: 識別子、調査区分（第1次／第2次）、関連する第1次調査への参照（第2次の場合）、家屋識別情報（住所・家屋番号・構造種別等）、状態（下書き/提出/承認/確定）、位置情報（緯度経度）、作成者、承認者、確定者、各種日時。
- **被災者情報 (PII)**: 氏名・連絡先等。アクセス制御・暗号化の対象。
- **被害入力 (Assessment Input)**: （第1次）外力・流失該当フラグ、傾斜実測値、浸水深。（第2次）外部・内部の部位別損傷率 等。
- **判定結果 (Assessment Result)**: 損害割合（％）、被害度区分（6区分）、計算根拠（経路別の中間値）。第1次・第2次それぞれの結果を保持。
- **正式判定 (Official Determination)**: 同一家屋で第1次・第2次の結果が併存する場合に、管理者が正式採用した区分とその選択者・日時。（出典: CQ-S4=B）
- **画像 (Photo)**: S3 オブジェクト参照、部位/全体の区分、撮影日時、（任意で）位置情報。
- **マスタ**: 部位定義（外部・内部）、構成比標準値、浸水深-損害割合換算表（運用指針準拠、固定／バージョン管理）。
- **ローカル下書き (Local Draft / IndexedDB)**: 同期前の入力内容・進捗状態・撮影画像（バイナリ）を端末ローカルに保持する。被災者 PII・画像を含むため §6（SECURITY-01）に従い保護・保持期間・同期後消去の対象とする。サーバ確定データとは別管理で、提出時に一括同期される。（出典: AD2-FU=A）

## 6. セキュリティ要件（SECURITY ベースライン適用）

個人情報（被災者情報）を扱う本番想定アプリのため、セキュリティ拡張を全面適用する。要件レベルで関連する主項目:

- **SECURITY-01**: PostgreSQL・S3 の保存時暗号化、通信の TLS 強制。**加えて、ローカルファースト採用に伴いブラウザ IndexedDB に保持する PII・画像を保護すること**: 端末ローカルの機微データはアプリ層で暗号化して保存し、保持期間を限定し、**提出・同期完了の確認後にローカルの PII・画像を確実に消去**する（喪失防止のため同期成功確認前は消去しない）。共用端末を想定し、認証セッション終了時のローカルデータ取り扱いを定義する。（出典: AD2-FU=A）
- **SECURITY-03/14**: 構造化ログ・監査ログ（PII/トークンを出力しない）、認証/認可失敗のアラート、ログ改ざん防止と保持期間。
- **SECURITY-04**: HTML 配信エンドポイントのセキュリティヘッダ（CSP 等）。既存 helmet を強化。
- **SECURITY-05**: 全 API 入力の検証（zod）。数値入力（浸水深・損傷率）は範囲・型を厳格に検証。
- **SECURITY-08**: デフォルト拒否の認可、オブジェクトレベル認可（他者の調査へのアクセス防止）、ロールに基づく機能レベル認可（承認/確定は管理者のみ）。
- **SECURITY-09**: デモ用 Task/User ドメイン・サンプルを削除（C2=A と整合）。本番エラーは詳細を秘匿。
- **SECURITY-10**: 依存のロックファイル/脆弱性スキャン、Docker の固定タグ。
- **SECURITY-13**: 確定データ・PII 変更の監査（実施者・日時・前後値）。
- **SECURITY-15**: フェイルクローズ、全外部呼び出しの例外処理、グローバルエラーハンドラ。

> 注: 公開ネットワーク公開時の認証・認可・レート制限・CORS 制限は設計段階で具体化する（SECURITY-08/11）。

## 7. プロパティベーステスト対象（PBT 全面適用）

被害度計算が中核ロジックのため、PBT を重点適用する（詳細プロパティは機能設計 PBT-01 で特定）。想定する代表的プロパティ:

- **不変条件 (PBT-03)**: 損害割合は常に 0–100% の範囲に収まる。損害割合 → 被害度区分のマッピングは境界値（10/20/30/40/50%）で正しく区分される（単調・網羅・排他）。
- **不変条件**: 部位別損傷率×構成比の合計は、各部位損傷率が全て 100% のとき住家全体 100% となる（構成比の総和=1 の整合）。
- **ラウンドトリップ (PBT-02)**: 調査データ・結果の シリアライズ/デシリアライズ（DTO 変換、CSV/PDF 入出力の構造化部分、**IndexedDB ローカル保存↔同期ペイロードの往復**）の往復同一性。
- **冪等性 (PBT-04)**: 同一入力に対する計算の再実行が同一結果（FR-27）。確定操作の冪等性。
- **状態 (PBT-06)**: 調査の状態機械（下書き→提出→承認→確定）における不正遷移の排除。
- フレームワークは fast-check（Vitest 連携）を想定（PBT-09、設計/NFR で確定）。

## 8. 制約・前提

- 既存 CATAPULT の技術構成・DDD ガイドライン（`docs/development-guidelines.md`）に従う。
- 計算ロジックの数値根拠は `docs/references/` の内閣府公式資料に準拠。外部ドメインからの資料取得は cao.go.jp / bousai.go.jp に限定。
- 対象災害は**水害のみ**。地震・風害・地盤被害は本リリースのスコープ外。

---

## 9. スコープ外（Out of Scope）

- 地震・風害・液状化等の被害認定（水害以外）。
- 罹災証明書そのものの発行・交付業務（本アプリは調査・判定と結果出力まで）。
- 構成比マスタのユーザー編集（標準固定値を使用、A5=A）。
- 大規模（数千人同時）対応のスケーリング設計（中規模を前提）。

---

## 10. 主要な受け入れ基準（抜粋）

- **AC-1**: 調査員が1家屋の調査を作成し、部位別の実測値・損傷率を入力すると、損害割合（％）と6区分の被害度が自動計算・表示される。
- **AC-2**: 浸水深・傾斜・外力・部位別損傷率の各判定経路が運用指針どおりに動作し、境界値（例: 損害割合 50%）で正しい区分になる。
- **AC-3**: 部位ごとおよび全体の画像を撮影・添付でき、調査に紐付く。
- **AC-4**: 状態遷移が権限に従う（調査員=提出まで、管理者=承認・確定、閲覧者=参照のみ）。不正な遷移・他者リソース操作は拒否される。
- **AC-5**: 確定済み調査の PDF 出力と、複数調査の CSV エクスポートができる。
- **AC-6**: 個人情報・確定データの変更が監査ログに記録され、保存時/通信時の暗号化が有効。
- **AC-7**: 計算ロジックに対する PBT（境界・不変条件）と例示ベーステストが整備され、CI で実行される。
- **AC-8**: 確定済みの第1次調査に対し、管理者または調査員が第2次調査（再調査）を開始でき、外部＋内部の全部位の損傷率入力から損害割合・6区分が算出される。第1次調査に関連付けて履歴が残る。
- **AC-9**: 同一家屋に第1次・第2次の結果が併存する場合、双方が併記表示され、管理者がいずれかを正式判定として選択・確定できる。
- **AC-10**: 調査入力時、進捗パンくず（ステッパー）に現在・完了・残ステップが表示され、標準順序（住民情報入力→1次/2次選択→調査入力→（該当時）2階部分の割合調整→計算結果確認）で案内される。該当しないステップはスキップ表示される。
- **AC-11**: 複数階建ての調査で、階ごとの床面積比率を調整すると、その按分が部位別損害割合の算定に反映され、計算結果が変化する。
- **AC-12**: オフライン（通信不通）状態でも調査員が入力・撮影を継続でき、入力内容・画像が IndexedDB に保持される。アプリ再読込後もローカルデータが復元される。「提出」操作でサーバへ一括同期され、同期成功の確認後にローカルの PII・画像が消去される。同期失敗時はローカルデータが保持され再試行できる。

## 11. 未確定事項・リスク（設計段階で具体化）

- 浸水深-損害割合の換算表、部位別構成比の正確な数値（運用指針 水害編・記入の手引きから抽出が必要。大判の「損傷程度の例示」資料は未取得）。
- 住家種別（木造・プレハブ／非木造）ごとの構成比の取り扱い範囲（A5=A は標準固定だが、水害でも木造/非木造で様式が異なる点の扱い）。
- 「2階部分の割合調整」の具体仕様（階ごとの床面積按分の算定方法・対象部位・既定値）を運用指針から確定する（Functional Design）。
- PWA 化・ローカルファーストの具体実装（IndexedDB スキーマ、ローカル暗号化方式、同期キュー・再試行・競合解決、Service Worker の範囲、同期後消去の確実性）を設計段階で確定する（NFR-02, FR-18/19, SECURITY-01）。
- 公開時の認証基盤・ネットワーク/レート制限の具体構成。
