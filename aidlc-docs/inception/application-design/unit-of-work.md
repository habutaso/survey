# ユニット・オブ・ワーク 定義（Unit of Work）

住家被害認定調査（水害）効率化アプリを、承認済み分解計画（`unit-of-work-plan.md`）に基づき **11 の開発ユニット**へ分解する。デプロイ形態は **UG1=A** に従い、単一 Docker コンテナーのモノリス内の**論理モジュール**として扱う（独立デプロイはしない）。

## 分解の根拠（回答反映）
- **UG1=A**: 単一デプロイのモノリス内の論理モジュールとして分解
- **UG2=B + UG2-FU=A**: 計算エンジンを第1次（U3a）/第2次（U3b）に分割し、共有要素を計算コア **U3c** に集約
- **UG3=B**: フロントエンドを「ローカルファースト基盤（U6f）」と「画面ジャーニー UI（U6u）」に分割
- **UG5=B**: 横断的関心事（監査・検証・暗号化）を専用ユニット **U-Cross** に独立
- **UG6=A**: 提出時一括同期 API（`ingestSubmission`）はサーバ側 U2 に内包
- **UG4-FU=A**: 構築順序は推奨順

---

## 1. ユニット一覧

### U0 — デモ削除・基盤整備
- **種別**: モジュール（横断・基盤整備）
- **責務**: 既存デモ `task` ドメイン・サンプル（controller/tests/UI）の削除、`user` ドメインのロール拡張準備、`common` 規約・ID（ULID 等）整備。リリース構成からデモ User/Task を除去。
- **主なコンポーネント**: `task`（削除）, `common`
- **関連ストーリー**: US-806（デモ削除部分）
- **関連要件**: SECURITY-09, C2

### U1 — 認証・ユーザー/ロール基盤
- **種別**: モジュール（Domain: 既存 `user` 拡張）
- **責務**: Cognito 連携認証、利用者解決（findOrCreateUser）、ロール（調査員/管理者/閲覧者）保持、認可基盤（`api/private/hooks` L1 ＋ Model 層 `assert*` L2 の土台）。
- **主なコンポーネント**: `auth`, `service/cognito`, `api/private/hooks`
- **関連ストーリー**: US-101, US-102, US-103, US-802
- **関連要件**: FR-40〜43 / SECURITY-08, AD6

### U-Cross — 横断（監査ログ・入力検証・暗号化）
- **種別**: モジュール（横断基盤）
- **責務**: 監査ログ基盤（状態遷移・PII/確定変更・認証認可失敗の記録）、入力検証基盤（zod バリデータの境界適用）、暗号化基盤（保存時・通信時・ブラウザローカル PII/画像のアプリ層暗号化ユーティリティ）、セキュアな既定動作（CSP 等ヘッダ・エラー秘匿・fail closed）。各ユニットが利用する共通サービス。
- **主なコンポーネント**: `audit`, `common/validators`, 暗号化ユーティリティ（サーバ＋Web Crypto）, セキュリティヘッダ/エラーハンドラ
- **関連ストーリー**: US-801, US-803, US-804, US-806（既定動作部分）
- **関連要件**: NFR-08 / SECURITY-01, 04, 05, 09, 13, 14, 15

### U2 — 調査管理（Survey 集約）＋提出時一括同期 API
- **種別**: モジュール（Domain: 集約ルート）
- **責務**: 1家屋=1調査の集約。家屋識別情報・被災者 PII・GPS・状態機械（下書き→提出→承認→確定）・調査区分参照・正式判定の保持と遷移。第1次/第2次の入力データ実体（`FirstSurveyEntity`/`SecondSurveyEntity`）保持。提出時一括同期の受け口 `POST /api/private/surveys/submission` → `ingestSubmission`（**UG6=A**）。
- **主なコンポーネント**: `survey`, `firstSurvey`, `secondSurvey`（データ実体）, `surveyUseCase`, `secondSurveyUseCase`
- **関連ストーリー**: US-201, US-202, US-203, US-207（サーバ側同期）, US-501〜505, US-601, US-605, US-606
- **関連要件**: FR-01〜09, FR-18, FR-19 / SECURITY-01, 13

### U3c — 計算コア（共有, assessment-core）
- **種別**: モジュール（Domain: 純粋計算カーネル / PBT 重点）
- **責務**: U3a/U3b が共有する計算要素を一元化。マスタ定数（構成比・浸水深換算表・部位定義・既定按分）、6区分マッピング `classifyDamageLevel`、階按分 `applyFloorRatio`、結果不変条件・PBT ハーネス、`AssessmentResult` 型。フレームワーク・永続化に非依存の純粋関数。
- **主なコンポーネント**: `assessment/constants`, `assessment/classifyDamageLevel`, `assessment/applyFloorRatio`, `assessment/types`, PBT ハーネス
- **関連ストーリー**: US-402, US-403, US-404（階按分・区分）, US-805
- **関連要件**: FR-24, FR-26, FR-27, FR-28, FR-30 / NFR-04 / §7 PBT / AC-2, AC-7

### U3a — 第1次被害度計算
- **種別**: モジュール（Domain: 純粋計算）
- **責務**: 第1次経路（外力・流失該当 / 傾斜実測 / 浸水深）からの損害割合（0–100%）を決定論的に算出する `calcFirst`。U3c に依存。
- **主なコンポーネント**: `assessment/calcFirst`
- **関連ストーリー**: US-301, US-302, US-303（計算反映）, US-401
- **関連要件**: FR-20, FR-21, FR-22, FR-25, FR-27 / NFR-04

### U3b — 第2次被害度計算
- **種別**: モジュール（Domain: 純粋計算）
- **責務**: 第2次経路（外部部位＋内部部位の損傷率×構成比、階按分）からの損害割合・6区分を算出する `calcSecond`。U3c に依存。
- **主なコンポーネント**: `assessment/calcSecond`
- **関連ストーリー**: US-602, US-603, US-604（計算反映）
- **関連要件**: FR-23, FR-24, FR-25, FR-27 / NFR-04

### U4 — 画像管理
- **種別**: モジュール（Domain）
- **責務**: 部位/全体写真の S3 連携、調査・部位・ステップへの紐付け、複数枚保持。
- **主なコンポーネント**: `photo`, `service/s3`
- **関連ストーリー**: US-304, US-305
- **関連要件**: FR-12, FR-13, FR-14 / SECURITY-01

### U5 — 結果出力・一覧
- **種別**: モジュール（Domain/Service 複合）
- **責務**: 調査票様式 PDF（単票）・CSV/Excel（複数件）のサーバ側生成（**AD4=A**）、調査の一覧・検索（認可範囲内）。
- **主なコンポーネント**: `export`, `exportUseCase`, 一覧/検索 Query
- **関連ストーリー**: US-701, US-702, US-703
- **関連要件**: FR-31, FR-32, FR-33 / NFR-03 / SECURITY-08

### U6f — ローカルファースト基盤（IndexedDB＋同期）
- **種別**: モジュール（Frontend: hooks/utils）
- **責務**: 入力内容・進捗・撮影画像を IndexedDB にオフライン保持・復元、アプリ層暗号化（Web Crypto）。提出時一括同期（`SyncOnSubmit`）＝送信・キューイング・再試行・同期成功確認後のローカル PII/画像消去。
- **主なコンポーネント**: `LocalDraftStore`, `SyncOnSubmit`/`syncService`, ネットワーク状態監視
- **関連ストーリー**: US-204, US-207（クライアント側）
- **関連要件**: FR-18, FR-19 / NFR-02 / SECURITY-01

### U6u — 画面ジャーニー UI
- **種別**: モジュール（Frontend: features/components）
- **責務**: ガイド付きステップ進行・進捗パンくず（ステッパー）・ステップ間移動・スキップ判定、各入力画面、計算結果表示・第1次/第2次併記、PDF/CSV 出力トリガ、ログイン UI。U6f を基盤として利用。
- **主なコンポーネント**: `SurveyWizard`, 入力ステップ群, `ResultView`, `ExportTrigger`, ログイン画面
- **関連ストーリー**: US-101〜103（UI）, US-205, US-206, US-301〜305（UI）, US-401, US-402, US-404（表示/入力）, US-605, US-701, US-702（トリガ）, US-703（UI）
- **関連要件**: FR-15, FR-16, FR-17, FR-30, FR-09 / US 各 UI

## 2. 構築順序（UG4-FU=A）

```
U0 → U1 → U-Cross → U2 → U3c → U3a → U3b → U4 → U5 → U6f → U6u
```

| 段階 | ユニット | 着手可能条件 |
|---|---|---|
| 1 | U0 | なし（最初に実施） |
| 2 | U1 | U0 完了 |
| 3 | U-Cross | U1 完了（認可基盤を前提） |
| 4 | U2 | U1, U-Cross 完了 |
| 5 | U3c | U-Cross 完了（型・PBT 基盤） |
| 6 | U3a | U3c 完了 |
| 7 | U3b | U3c 完了 |
| 8 | U4 | U2 完了 |
| 9 | U5 | U2, U3a/U3b 完了 |
| 10 | U6f | U2 の同期 API 確定 |
| 11 | U6u | U6f, 各 API 確定 |

> クリティカルパス: U0 → U1 → U-Cross → U2 → U3c → (U3a/U3b)。U4/U5 は U2 と計算ユニットに依存。U6f/U6u はサーバ API 確定に追従（依存が許す範囲で並行着手可）。

## 3. ユニット境界の検証
- **凝集性**: 各ユニットは単一の境界づけられたコンテキストまたは明確な技術的関心事に対応。
- **計算の純粋性**: U3c/U3a/U3b は永続化・フレームワークに非依存（`assessment` 純粋カーネル, AD1=A）。
- **横断の一元化**: 監査・検証・暗号化は U-Cross に集約し重複を回避（UG5=B）。
- **重複回避**: 計算の共有要素は U3c のみに存在（UG2-FU=A）。
- **デプロイ単一性**: 全ユニットは 1 コンテナーに同梱（UG1=A）。
