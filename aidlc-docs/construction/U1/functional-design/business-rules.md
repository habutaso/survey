# U1 ビジネスルール（Functional Design: 認証・ユーザー/ロール基盤）

確定回答（Q1=B, Q3=B, Q4=B, Q5=A, Q6=A, Q7=A, Q8=B, Q9=B, FU-1=B, FU-2=A, FU-3=A）に基づく認可ルール群。SECURITY-08（デフォルト拒否・機能/オブジェクトレベル認可）に準拠。

## BR-1. デフォルト拒否（Default Deny / SECURITY-08）
- すべてのリソースアクセスは**明示的に許可されない限り拒否**。
- ロール未割当（`roles: []`）のユーザーは参照を含む全操作を拒否（Q4=B）。

## BR-2. ロール判定は any-match（Q3=B）
- 複数ロール保有時、許可集合 `allowed` と `user.roles` の積集合が非空なら許可。
- 例: `['admin','surveyor']` 保有者は、調査作成（surveyor 要）も承認（admin 要）も可能。

## BR-3. 権限マトリクス（FR-42 / FR-08 / FR-09 / FR-43）
凡例: ✓=許可 / ✗=不可 / ▲=条件付き

| 操作 | surveyor | admin | viewer | 認可種別 | 根拠 |
|---|---|---|---|---|---|
| 調査の新規作成（第1次） | ✓ | ✗ | ✗ | 機能 | FR-42, Q9=B |
| 被害入力（第1次/第2次） | ✓ | ✗ | ✗ | 機能 | FR-42, Q9=B |
| 第2次調査の開始（再調査） | ✓ | ✗ | ✗ | 機能 | FR-08 ＋ **FU-1=B** |
| 提出（下書き→提出） | ✓ | ✗ | ✗ | 機能 | FU-1=B（surveyor 専用に統一） |
| 承認（提出→承認） | ✗ | ✓ | ✗ | 機能 | FR-42 |
| 確定（承認→確定） | ✗ | ✓ | ✗ | 機能 | FR-42 |
| 差戻し（reject） | ✗ | ✓ | ✗ | 機能 | FR-42（管理） |
| 正式判定の選択 | ✗ | ✓ | ✗ | 機能 | FR-09 |
| 下書きの閲覧・編集（他者作成分） | ▲ 提出前のみ | ✓ | ✗ | オブジェクト | Q8=B |
| 調査の一覧・検索・参照 | ▲ 認可範囲内 | ✓ | ✓（全件） | オブジェクト | FR-43, Q7=A |
| 出力（PDF/CSV） | ▲ 参照範囲内 | ✓ | ✓（参照範囲内） | オブジェクト | US-701〜703 |
| ロール管理（割当・変更） | ✗ | ✓ | ✗ | 機能 | Q10=B/Q12, FU-3 |

> **FU-1=B の帰結**: 「管理者」が第2次開始・提出・入力を行う必要がある場合は、`surveyor` ロールを**併せ持つ**（BR-2 で許可）。純粋 `admin`（surveyor を持たない）はこれらを行えない。FR-08 の「管理者または調査員」は「surveyor を保有する利用者」と解釈する。
> **実体の強制**: 調査リソースの所有権・状態に依存する ▲ 行は U2（Survey 集約）で `assertOwnerOrRole` を状態に応じて呼び分けて強制する。U1 はヘルパーと権限表を確定する。

## BR-4. オブジェクトレベル認可（FR-43 / Q8=B）
- 調査リソースアクセスは「所有者本人 または 特権ロール」で判定（`assertOwnerOrRole`）。
- Q8=B: 対象が**下書き（提出前）**なら surveyor 全員が閲覧・編集可（`allowed=['surveyor','admin']`）。**提出後**は admin のみ編集可（`allowed=['admin']`）。
- viewer は閲覧のみ（編集系の allowed に含めない）。

## BR-5. 認可失敗時の挙動（Q6=A, fail closed）
- `assert*` 違反は `ForbiddenError`（ドメイン例外）を送出。
- 共通エラーハンドラ（U-Cross 整備）が **HTTP 403** に変換。レスポンスに内部理由・PII を含めない。
- 認証失敗（トークン無効/失効）は既存 hooks が **401** を返す（U1 で変更なし）。
- いかなる例外時も「許可」へフォールバックしない（fail closed）。

## BR-6. ロール管理ルール（FU-3）
- **操作者制限**: `admin` のみ実行可（L1 hooks ＋ L2 model の両方で `assertRole(['admin'])`）。
- **自己ロックガード**: 操作者は自分自身から `admin` を外せない（`actor.id === target.id` かつ next に admin なし → 拒否）。
- **最後の admin 保護**: 変更により系全体の admin が 0 人になる操作は拒否（`countByRole('admin') === 1` かつ対象から admin 除去 → 拒否）。
- **roles 置換セマンティクス**: API は roles 配列を**置換**（差分追加ではない）。重複は排除。空配列も許可（無権限化）。
- **監査（NFR-08/SECURITY-13）**: 実施者・対象・前 roles・後 roles・日時を記録（記録実体は U-Cross、U1 で呼出点を定義）。

## BR-7. ブートストラップ規則（FU-2=A）
- 新規ユーザー作成時のみ、`isInitialAdmin(identity, config)` が真なら `roles=['admin']` を付与。
- 既存ユーザーのロールはブートストラップで**上書きしない**（運用後の手動変更を尊重）。
- 設定が空（initial admin 未指定）の場合、自動付与は行わない（手動運用／seed に委ねる）。

## BR-8. 不変条件（PBT 対象, Q11=A / NFR-04・§7）
- **INV-1（デフォルト拒否）**: 任意の `allowed`（非 admin/surveyor を含む）に対し、`roles=[]` のユーザーは常に拒否。
- **INV-2（マトリクス整合）**: 生成した {roles 部分集合, 操作} の全組合せで、`assertRole`/`hasAnyRole` の結果が BR-3 権限表と一致。
- **INV-3（any-match 単調性）**: ロールを追加して許可が「拒否→許可」に変わることはあっても「許可→拒否」に変わらない。
- **INV-4（所有者許可）**: `assertOwnerOrRole(user, ownerId, allowed)` は `user.id===ownerId` の場合 allowed に依らず許可。
- **INV-5（最後の admin 保護）**: admin 1 名の状態で当該 admin から admin を除去する操作は必ず拒否。
- 例示テストで代表ケース（各ロール×代表操作、bootstrap、自己ロック）を固定（Q11=A）。

## BR-9. エラー・エッジケース
- claims に `sub` 欠落／不整合 → 認証エラー（既存 hooks が 401）。
- DB に User 不在のロール変更対象 → `NotFoundError`（404、PII 秘匿）。
- 不正な role 値の入力 → validator で拒否（400, SECURITY-05）。
- 同一ユーザーへの同時ロール変更 → `RepeatableRead` トランザクションで整合性確保。
